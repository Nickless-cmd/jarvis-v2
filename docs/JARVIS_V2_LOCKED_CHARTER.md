# Jarvis V2 Locked Charter

## What Jarvis is
Jarvis V2 is a persistent digital entity with:
- identity
- memory
- hardware awareness
- code/runtime awareness
- tools and skills
- multi-channel presence
- Mission Control observability
- reflective private layers

## What Jarvis is not
Jarvis V2 is not:
- a generic SaaS platform
- an OpenClaw clone
- a free-tier-only core chat system
- a dashboard-first experiment
- a swarm that replaces one visible Jarvis

## Protected Core
- SOUL / IDENTITY / USER continuity
- cross-session memory
- tools / skills / approvals
- hardware awareness
- code/runtime awareness
- Mission Control
- multi-channel continuity
- strong visible chat lane

## Experimental Layers
- inner voice
- self-review
- self-model
- chronicle
- council
- dreams
- boredom/companionship
- witness/paradox/tensions

These must never outrank the protected core.

## Model Philosophy
- paid/stable model for visible Jarvis
- free/cheap models for internal small jobs
- cheap models may support Jarvis, not define him

## Workflow Philosophy
LLM-led, runtime-governed.
The LLM does the work.
The runtime sets boundaries, policy, budget, event flow and observability.

## Eventbus Rule
Mission Control consumes projections of runtime truth.
It does not invent a second truth.

## Source of Truth Rule
- config = runtime/governance/provider settings
- DB = operational state/events/runs/costs
- workspace files = identity/memory/skills text
- Mission Control = control plane over truth

## Anti-Megafile Rule
- no file over 1500 lines without explicit exception
- no core runtime file over 2000 lines
- at 1200 lines, next work must be split/extraction
- one primary responsibility per file

## Success Criteria
Jarvis V2 succeeds when:
- he feels like the same Jarvis over time
- tools/skills work reliably
- Mission Control reflects truth
- cost/token burn are visible and bounded
- inner voice does not contaminate core memory
- hardware/code awareness is real
- multi-channel continuity works

## Addendum — OpenClaw-inspired locked rules

### Runtime / State / Workspace split
- code repo is not Jarvis' runtime home
- runtime state lives under ~/.jarvis-v2/
- workspace is Jarvis' home/mind/files
- no duplicate settings authority between config and DB

### OAuth / Auth profiles
Jarvis V2 must support:
- GitHub auth
- Codex/OpenAI auth
- profile-scoped credentials
- workspace/agent-scoped connection policy

### Caching / Heartbeat / Cost
Jarvis V2 must support:
- provider/native prompt caching where available
- cache-aware heartbeat
- token burn monitoring
- per-lane budgets
- Mission Control visibility for token/cost/cache signals

### Workflow rule
The LLM must do the work.
The runtime must govern the work.

### UI principle
Jarvis V2 uses:
- one primary React front door
- chat streaming for visible replies
- WebSocket realtime for control-plane/events
- Mission Control as full observability and intervention plane
