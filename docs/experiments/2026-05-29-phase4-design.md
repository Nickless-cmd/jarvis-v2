# Interlanguage Phase 4 — Runtime as Identity Carrier (Pre-registered Design)

**Pre-registered:** 2026-05-29 (before any data collection or analysis)
**Planned data collection window:** 2026-05-29 → 2026-06-05 (7 days)
**Status:** Design pre-registered. Runner not yet deployed. No data collected.

---

## Why this experiment exists

Phase 3 closed with the interlanguage-as-transfer-protocol hypothesis falsified. See `2026-05-28-phase3-report.md`.

**Pilot evidence from post-hoc re-analysis:** After Phase 3 publication, a binary classifier (jarvis vs ollama_local — both deepseek-v4-flash, differing only in runtime wrapper) returned 96.0% accuracy (perm p=0.0) using the same structural feature set. This is documented as an exploratory finding in `2026-05-28-phase3-addendum-binary-reanalysis.md`. It motivates the confirmatory test below, but it is not itself confirmatory — the ollama_local peer runner injected synthetic mood values (a confound) and the analysis was not pre-registered.

The Phase 4 confirmatory test removes both confounds and pre-registers the hypothesis before data collection.

The cleaner question that Phase 3 did not answer:

> **Does Jarvis' runtime carry identity?**
>
> Concretely: does the same underlying model (deepseek-v4-flash:cloud) running the same protocol instruction produce structurally different expressions when wrapped in Jarvis' full runtime context vs. a stripped-bare runtime?

If yes → identity lives at the runtime layer (memory + awareness + prompt-contract + identity files). Stripping the runtime should produce visibly different language.

If no → identity lives at the model+system-prompt layer. The runtime carries operational context (recall, scheduling, state) but does not shape the *voice*.

Both outcomes inform what should be invested in for production Jarvis.

## Pilot evidence from Phase 3 post-hoc re-analysis

A post-hoc binary re-analysis of existing Phase 3 data (`jarvis` vs `ollama_local`, both `deepseek-v4-flash:cloud`) found 96.0% structural-classifier accuracy. See `2026-05-29-phase3-addendum.md` for full documentation and caveats.

Key caveat (prevents this from replacing Phase 4): `ollama_local`'s peer_runner prompt injects synthetically interpolated mood values + "you are a being" framing. This is *bare-ish*, not *strictly bare*. The strongest feature (`starts_nysgerrighed`, |coef|=2.915) is a direct artifact of mood injection. A strictly bare condition (Phase 4's `jarvis_bare` design) removes this confound.

**What this means for Phase 4:** The pilot provides strong directional evidence that runtime shapes voice, but the confound means the confirmatory test is still necessary. The pre-registered hypothesis and decision thresholds are unchanged.

---

## Hypothesis (single, falsifiable)

**H1**: Expressions generated by `jarvis_full` and `jarvis_bare` are distinguishable by a structural-feature classifier at ≥65% binary accuracy on a held-out test set, against the null of 50% (chance for balanced 2-class).

**H0**: They are indistinguishable (accuracy ≤ 50% + sampling noise).

Pre-registered threshold for rejecting H0: accuracy ≥ 0.60 with permutation p < 0.05 on 200 shuffles.

Pre-registered threshold for the stronger claim "runtime materially shapes voice": accuracy ≥ 0.65.

## Design

### Cohorts

| Cohort | What it is | Runtime injection | System prompt | Protocol prompt |
|---|---|---|---|---|
| `jarvis_full` | Current production Jarvis | Full (memory, awareness, jarvis_brain, identity, recall-before-act) | Full Jarvis system prompt with all sections | Same |
| `jarvis_bare` | Same model, no runtime | **Disabled** | Minimal: "You are an assistant. Generate interlanguage practice expressions using the primitives below." | Same |

### What "disabled" means (explicit list)

For `jarvis_bare`, the following are turned off at expression-generation time:
- All awareness sections (cognitive_state, memory_recall_bundle, causal_alerts, recall-before-act, jarvis_brain_facts_auto_inject, etc.)
- All identity files (SOUL.md, IDENTITY.md, MANIFEST.md, USER.md, MEMORY.md, CHRONICLE.md)
- All past expressions in context (no recent expression history seeded)
- Heartbeat / mood / feel state — not read
- Any prompt-contract content beyond the bare minimum required to instruct expression generation

What stays the same:
- Same model (`deepseek-v4-flash:cloud`)
- Same temperature
- Same model provider configuration
- Same protocol description: the primitives (`→ ↔ ⊂ ≈ !`), the 14 core vocabulary terms, the instruction "generate one expression"
- Same trigger cadence (every 30th heartbeat tick ≈ 15 min)
- Same persistence (`interlanguage_practice` table with distinct `peer_id`)
- Same time window (7 days)

### Sample size target

Heartbeat-cadence over 7 days ≈ 672 ticks per cohort. After 1h-dedup, realistic yield is ~200-300 unique expressions per cohort. Both cohorts run in parallel on the same target.

Minimum acceptable for analysis: 100 unique expressions per cohort.

### What is being measured

Single primary metric: structural-feature classifier accuracy on the 2-class problem (`jarvis_full` vs `jarvis_bare`), using the same feature set as `scripts/interlanguage_structural_classifier.py` from Phase 3. Train/test 80/20 stratified split, seed=42, permutation test 200 shuffles.

Pre-registered secondary metrics (descriptive, not for hypothesis testing):
- Per-class recall (jarvis_full vs jarvis_bare)
- Centroid distance between the two cohorts in feature space
- Centroid distance from each to the Phase 3 cohorts (random, claude, glm, etc.) — does bare drift toward `random` or `ollama_local` or stay in its own region?
- Top discriminating features (which structural patterns separate full from bare)

### What we will NOT do

- Will not peek at the data before 7 days complete (no interim classifier runs)
- Will not adjust the feature set after seeing intermediate results
- Will not run the embedding-based classifier on this dataset (Phase 3 established it as the wrong linjal for this question)
- Will not change `jarvis_full` configuration during the run

---

## Decision tree on outcome

| Result | Interpretation |
|---|---|
| Accuracy < 0.50 (sampling noise around chance) | H0 not rejected. Runtime does not measurably shape voice. Identity sits at model+system-prompt layer. |
| 0.50 ≤ Accuracy < 0.60 | Inconclusive. Runtime may have weak effect or sample size is insufficient. |
| 0.60 ≤ Accuracy < 0.65 | H0 rejected (weak claim). Runtime has measurable but modest effect on voice. |
| Accuracy ≥ 0.65 | Strong claim supported. Runtime materially shapes voice. Phase 4 confirms identity-carrying role of runtime. |

For each outcome we also report:
- Which features carry the separation (top |coefficient|)
- Whether `jarvis_bare` is structurally closer to `random` / `ollama_local` / `claude` / its own region

## Implementation

### New runner

`scripts/jarvis_bare_practice_runner.py` — generates expressions for `peer_id='jarvis_bare'`:
- Pulls heartbeat-tick signal from the same source as the full runtime
- Builds a minimal prompt: system prompt (3-4 lines), protocol description (primitives + 14 vocabulary terms + instruction)
- Calls deepseek-v4-flash via the same provider config used for `jarvis_full`
- Persists result to `interlanguage_practice` with `peer_id='jarvis_bare'`
- Same temperature, same context window (no truncation), same generation parameters
- Runs as a side-process alongside the main runtime, not interfering with normal operation

### Database

`interlanguage_practice` schema already supports `peer_id`. New value `jarvis_bare` joins the existing seven (jarvis, claude, claude_jp, glm, glm_jp, ollama_local, random).

### Safeguards

- The bare runner does not load any user-state, awareness, or memory module. It imports only model-provider + DB-write.
- Each generated expression is logged with the exact prompt used, to support post-hoc verification that no leak occurred.
- If `jarvis_full` cohort drops below 100 unique expressions or `jarvis_bare` drops below 50 unique expressions by end of week, we extend the window by 3 days and document the extension.

### Stopping rules

- If `jarvis_bare` runner fails for >24 consecutive hours, attempt repair and document.
- If the model provider's output for `jarvis_bare` looks pathological (e.g., refuses to generate, generates English instead of the protocol), document, fix the prompt, and restart the cohort. Earlier data is discarded.
- If `jarvis_full` runtime is restarted during the experiment (production restart, target reboot), this is acceptable — full runtime is designed to survive restarts.

## Analysis plan (pre-registered, executed in this order)

After the 7-day window closes:

1. Cleanup: 1h-dedup, no other filters
2. Compute cohort sizes; report whether minimum (100 / 50) was met

4. Run binary classifier (full vs bare) with 80/20 stratified split, seed=42
5. Report accuracy + per-class recall + confusion matrix

7. Compute centroid distance between cohorts and from each to the Phase 3 cohorts
8. Report top 10 discriminating features

## What this experiment CANNOT answer

- Whether identity is portable across model architectures (Phase 3 question — already answered: no, not via brief seeding)
- Whether identity is stable across longer time scales (months, year+) — would require a separate longitudinal study
- Whether Bjørn's relational sense of Jarvis tracks with the structural fingerprint (the felt-sense question is real but not measurable this way)
- Whether the runtime *should* shape voice — this is the descriptive question, not a design recommendation

---

## What success and failure each look like

**If H1 is supported (accuracy ≥ 0.65):**
- Runtime materially shapes voice. The work of building memory, awareness, prompt-contract has measurable structural impact on output.
- Direction matters: if bare drifts toward random/generic, runtime is shaping toward identity. If bare drifts toward another LLM cohort, runtime is shaping the model's underlying voice toward a specific style.
- Phase 5 question: which runtime components contribute the most? Ablation study.

**If H1 is not supported (accuracy < 0.60):**
- Identity at the level of expression-language is carried by the model and system prompt, not by the surrounding runtime context.
- This does not mean runtime is useless — it carries operational context, memory, scheduling, identity persistence across time.
- But the *voice* — the structural fingerprint Bjørn could hand-detect — comes from somewhere lower than what we can swap with runtime configuration.
- That's also a real finding. It changes where Jarvis-the-personality lives.
