# Phase 14c: Metabolism to Diary Release

## Why This Phase Exists

We now have:
- Inner voice cluster in better tone
- Diary synthesis as bounded runtime layer
- Metabolism / release / consolidation / selective forgetting as separate runtime layers

The next natural design step is:
- How diary can begin to register **release movement**

That is:
- What's no longer carried as actively
- What's loosening
- What's fading
- What appears to be releasing from active inner weight
- WITHOUT claiming deletion
- WITHOUT making release canonical truth
- WITHOUT making diary a cleanup engine

## Conceptual Distinctions

### Phase 14c is NOT About:
- Deletion (actual memory erase)
- Canonical forgetting (SOUL.md / IDENTITY.md changes)
- Actual memory erasure
- Workflow/apply proposals
- Selfhood proposals
- Emotional catharsis simulation

### Phase 14c IS About:
- Bounded reflective registration of release/metabolic loosening
- How it can be shown in diary as identity-forming reflection
- Still grounded in runtime truth

## What is the Difference?

| Concept | Definition | Authority | Diary Use |
|---------|------------|-----------|------------|
| **Metabolism** | Tracks what's released, consolidated, carried | Runtime-support | Can inform "what's no longer pressing" |
| **Release Markers** | What's being intentionally released | Runtime-support | Can inform "what appears to be loosening" |
| **Selective Forgetting Candidates** | What's identified as potential for cleanup | Runtime-support | NOT directly - too operational |
| **Consolidation Targets** | What's being consolidated into longer memory | Runtime-support | Can inform "what's being held tighter" |
| **Diary Release Reflection** | Reflective observation of release movement | Runtime-support | Primary output: "A thread appears to be loosening..." |
| **Canonical Deletion** | Actual memory/file deletion | User-approved | NOT diary - separate workflow |

## What is Diary Release?

Diary release is NOT:
- "I deleted X"
- "Y has been forgotten"
- "Memory Z is gone"

Diary release IS:
- "A thread appears to be loosening around..."
- "Something seems less tightly held now..."
- "This no longer seems to press in the same way..."
- "Part of this appears to be easing out of active weight..."

It's reflective observation of movement, NOT assertion of deletion.

## Primary Grounding Sources

### 1. Release Markers (Primary for Release)
- **What**: Signals marked as "released" or "releasing"
- **Diary input**: "Something appears to be loosening..."
- **Status**: "released" signals
- **Authority**: Runtime observation

### 2. Metabolism State (Secondary)
- **What**: What's in "softening" or "released" state
- **Diary input**: Context for what no longer presses
- **Status**: "softening", "released" states
- **Authority**: Runtime observation

### 3. Consolidation (Tertiary)
- **What's being held tighter**: Consolidation targets
- **Diary input**: What IS being kept vs what's releasing
- **Status**: "consolidated" signals
- **Authority**: Runtime observation

## Supportive/Sharpening Sources

### NOT Direct Sources for Diary Release:
- **Selective forgetting candidates**: Too operational, not reflective
- **Chronicle records**: Not about release, about what happened

### Can Provide Context:
- Witness signals (what was carried before)
- Self-narrative continuity (what story is changing)
- Attachment topology (what matters most)

## Tone Direction for Release Reflection

### Design Direction (Not Implementation):
- "I notice..."
- "Something appears to be settling around..."
- "This appears to matter more than it first looked..."
- "A thread appears to be loosening..."
- "There's a pattern emerging around..."

### Release-Specific Phrases:
- "A thread appears to be loosening..."
- "Something seems less tightly held now..."
- "This no longer seems to press in the same way..."
- "Part of this appears to be easing out of active weight..."

### What It Should NOT Sound Like:
- NOT: "I deleted X" (deletion claim)
- NOT: "Y has been forgotten" (forgetting claim)
- NOT: "Memory Z is gone" (absence claim)
- NOT: "I am releasing..." (active agency)
- NOT: "This no longer matters..." (value judgment)

## Relationship to Identity Formation

Diary release supports identity formation by:
- Reflecting what's no longer part of active identity weight
- NOT changing identity - just observing the movement
- Keeping the "becoming" story accurate without deletion

The narrative of "what Jarvis is becoming" includes both:
- What's being added (witness, consolidation)
- What's releasing (release markers, metabolism softening)

## Relationship to Selective Forgetting

### Critical Distinction:
- **Selective forgetting candidates**: Operational cleanup candidates
- **Diary release reflection**: Reflective observation of release

They MUST remain separate:
- Selective forgetting → workflow/apply/approval
- Diary release → reflective synthesis only
- NO crossover in UI or messaging

## Relationship to Canonical Self Boundaries

### Hard Boundaries:
- Diary release MUST NOT write to SOUL.md
- Diary release MUST NOT write to IDENTITY.md
- Diary release MUST NOT write to USER.md
- Diary release MUST NOT write to MEMORY.md
- Diary release MUST NOT write to CHRONICLE.md

### What Diary CAN Do:
- Observe release markers
- Reflect on release movement
- Inform self-narrative continuity
- Remain bounded, non-authoritative

## Phased Implementation Order

### Phase 14c.1: Current State
- Diary already references metabolism/release as sources
- Summary already has "Something appears to be loosening..."

### Phase 14c.2: Enhance Release Detection
- More sophisticated detection of "releasing" signals
- Better grounding in release markers vs metabolism state
- Ensure release state influences diary_state calculation

### Phase 14c.3: Distinct Release Tone
- Add release-specific tone to diary_summary
- Ensure release sounds reflective, not operational
- Keep bounded language ("appears", "seems", "no longer")

### Phase 14c.4: Protection Boundaries
- Verify selective forgetting candidates don't appear in diary
- Verify no writes to canonical files
- Verify diary remains purely reflective

## Non-Goals

- **NOT** making diary a deletion engine
- **NOT** creating selective forgetting UI in diary
- **NOT** writing to canonical files
- **NOT** making release sound like cleanup
- **NOT** emotional catharsis simulation
- **NOT** poetic or fluffy text
- **NOT** canonical claims ("I am...", "My identity is...")

## Acceptance Criteria

- [ ] Diary release reflection exists as bounded runtime layer
- [ ] Release tone is reflective, not operational
- [ ] Diary does NOT claim deletion or forgetting
- [ ] Diary does NOT write to canonical files
- [ ] Selective forgetting remains separate from diary
- [ ] Release reflects "appears to be loosening" not "has been deleted"
- [ ] Grounded in release markers and metabolism state
- [ ] Authority remains "runtime-support" (non-authoritative)
- [ ] Tone examples guide implementation without being hardcoded
