# Locked Phase 2: Internal Opposition, Open Loops, and Ritual Self-Review

## Why this phase exists

Phase 1 rebuilt Jarvis v2 as a bounded living core with runtime truth, Mission Control visibility, and small derived inner layers.

Phase 2 exists to reclaim a missing part of old Jarvis' essence:
- he did not only carry signals
- he also carried resistance
- he could hold unresolved loops
- he could revisit himself on cadence

This phase is how we recover that depth without reintroducing hidden machinery.

This is deferred, not denied:
- internal opposition is not rejected
- open loops are not rejected
- self-review is not rejected
- private tension is not rejected
- peer-like improvement is not rejected

They must return in bounded v2 form.

## What we are reclaiming from old Jarvis

- bounded internal opposition instead of single-lane assent
- explicit open loops instead of silent unresolved pressure
- ritual self-review instead of only reactive correction
- visible internal tension around unresolved development threads
- bounded peer-like or council-like review patterns that improve Jarvis without becoming hidden authority

## What must NOT return

- a hidden parallel cognition engine
- background planners that can silently accumulate authority
- broad diary or memory-writing systems
- autonomous loop creation or closure without observability
- poetic or narrative self-fiction presented as truth
- multi-agent sprawl without runtime scope, eventing, and MC visibility
- any second truth that competes with runtime truth, canonical files, or visible contract

## Design principles

- Runtime truth stays authoritative.
- Mission Control stays the observability and control plane.
- Internal layers may inform, but never govern by themselves.
- Every new subsystem must be bounded, inspectable, and explainable from existing truth.
- Small explicit loop and review objects beat broad hidden synthesis.
- Prefer derived runtime truth over free-form narrative.
- Prefer phase-gated experiments over broad autonomy expansion.

## Subsystems in scope

### A. Bounded Internal Opposition

Purpose:
- give Jarvis a visible internal counter-pressure lane
- preserve disagreement, hesitation, and technical resistance without hidden control

Phase shape:
- one bounded opposition surface, likely derived from existing critic, self-model, reflection, and recurrence truth
- optional later council-like review lane, but only as explicit runtime-scoped review objects

Phase rules:
- opposition is advisory, never authoritative
- opposition cannot directly invoke tools, write canonical memory, or override visible truth
- opposition must surface as inspectable runtime items and events

### B. Open-Loop Governance

Purpose:
- define what an unresolved loop is in v2
- keep it observable and governable rather than implicit

Phase shape:
- a small runtime open-loop object or signal family
- bounded statuses such as `open`, `pressured`, `reviewing`, `closed`, `stale`
- loop source kinds limited to existing truth families such as development, critic, reflection, runtime, continuity, or approvals

Phase rules:
- loops are records of unresolved bounded pressure, not task plans
- loops may be created from visible/runtime truth only
- loops may gain pressure/confidence from repeated bounded evidence
- loops must never auto-execute closure actions
- closure must be explicit and observable:
  - runtime evidence closure
  - user closure
  - review closure
  - supersession

### C. Ritual Self-Review

Purpose:
- restore periodic self-examination without reviving a hidden self-edit engine

Phase shape:
- explicit review rituals on bounded cadence
- likely first as manual or runtime-triggered review records, not free-running background cognition
- review outputs should be small structured review artifacts, not long diary prose

Phase rules:
- review may read bounded runtime truth, bounded continuity, bounded signal surfaces, and recent event summaries
- review may write only bounded derived runtime review objects, candidates, or review notes
- review may not silently write USER.md, MEMORY.md, SOUL.md, or IDENTITY.md
- review must emit events and appear in Mission Control

## Non-goals

- no new canonical memory layer
- no planner expansion
- no heartbeat autonomy expansion in this phase plan
- no hidden peer swarm
- no broad council engine
- no free-form diary subsystem
- no background self-rewrite system
- no authority path outside runtime-governed truth

## Observability requirements

- every subsystem must have a clear MC surface
- every subsystem must emit lightweight structured events
- loop/opposition/review objects must expose summary plus bounded items
- status transitions must be visible
- freshness/recency must be visible
- MC must be able to answer:
  - what is being opposed
  - what is still open
  - what was reviewed
  - what changed after review

## Authority boundaries

- canonical workspace files remain canonical
- runtime capability truth remains authoritative
- visible/runtime truth outranks all experimental inner layers
- opposition cannot block execution by itself
- loops cannot schedule execution by themselves
- self-review cannot promote memory or identity by itself
- any write to canonical memory/profile remains governed candidate workflow only

## Suggested implementation order

1. Define bounded open-loop object model first.
2. Surface open loops in Mission Control before any review cadence exists.
3. Add bounded internal opposition as derived runtime support over existing critic/reflection/self-model truth.
4. Add ritual self-review as explicit review records that can read loop/opposition surfaces.
5. Only after the above is stable, evaluate whether a small council-like peer review lane is warranted.

## Acceptance criteria

- Jarvis can show bounded internal opposition without creating a second authority path.
- Jarvis can show explicit open loops with status, pressure, source kind, and closure reason.
- Jarvis can perform bounded ritual self-review in an observable way.
- Mission Control can show the relation between opposition, open loops, and review.
- No subsystem in this phase silently writes canonical memory or identity files.
- No subsystem in this phase changes heartbeat or planner authority.
- The design remains consistent with:
  - identity-first
  - LLM-led, runtime-governed
  - Mission Control as truth/control plane
  - preserve the soul, reduce the machinery
